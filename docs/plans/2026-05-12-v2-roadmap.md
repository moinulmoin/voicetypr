## V2 Product Direction

V2 should make VoiceTypr a reliable transcription layer that can be used from the desktop app, a weak-client/strong-host network setup, and CLI/tooling workflows.

The core idea is one shared transcription contract, not separate implementations for every surface.

## Phase 0: Network Sharing Integration

Network Sharing is now integrated into this V2 branch and remains a first-class V2 pillar. Its purpose is to let a weaker computer use a stronger machine for local network transcription.

Completed:

- Network-sharing implementation merged into `plan/v2-roadmap`.
- Automated frontend checks passed after merge conflict resolution.
- Automated backend tests passed after merge conflict resolution.
- Review fixes removed cleartext password persistence/IPC exposure, tightened lock scope, and cleaned up status/UI regressions.

Remaining release validation:

- Manually test strong-host and weak-client flow:
  - host starts sharing
  - client discovers/connects
  - client records audio and receives transcript
  - saved password is preserved across edit/restart
  - password removal changes auth behavior
  - offline server state is truthful
  - auth failure is clear
  - server-busy state is clear
## Phase 1: Shared Transcription Contract

Create one backend-level contract used by:

- desktop app recording
- network-sharing server
- future CLI
- future agent/tool interface

Inputs:

- audio file or recorded buffer
- engine/model selection
- language and translate settings
- output format
- optional post-processing profile

Outputs:

- transcript text
- optional structured segments/metadata
- engine/model used
- timing and performance metadata
- stable error codes

This contract should be the boundary before expanding V2 features.

## Phase 2: Network Sharing V2

Keep this focused on weak-client/strong-host usage.

Required:

- strong-machine host mode
- weak-client connect mode
- auth/token flow
- health/status endpoint
- model availability reporting
- remote readiness state that does not lie when offline
- clear client errors for offline/auth/server-busy/model-missing cases

Avoid:

- internet/cloud relay
- accounts
- collaboration features
- multi-tenant complexity

## Phase 3: CLI And Tool Interface

Add a CLI that uses the same shared transcription contract.

Target commands:

- `voicetypr status`
- `voicetypr models`
- `voicetypr transcribe --file audio.wav --json`
- `voicetypr transcribe --server host:port --file audio.wav --json`
- `voicetypr record --until-silence --json`

The important use case is agent/tool integration: Codex, Claude, scripts, and local automations can call VoiceTypr and receive clean text or JSON.

## Phase 4: Writing Intelligence

Build from the simplified and robust V2 plans.

Include:

- profiles
- custom vocabulary
- word replacements
- snippets
- context-aware cleanup
- deterministic fallback when AI/post-processing fails
- privacy-safe observability

Do not revive:

- the old 11-processor architecture
- a separate self-correction processor
- a separate voice-command processor
- heavy debug dashboards
- edit-script frameworks before the basic contract is stable

## Phase 5: Reliability And Release

- packaged Windows test if Windows remains in scope
